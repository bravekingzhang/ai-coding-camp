# Spec 分级标准（四层）

> 本页是 c1-05、c2-01 等分级测验的**唯一判分依据**。分级口径与测验答案哈希强绑定，逐条对齐，请不要凭感觉判。

不是所有改动都值得写一份完整的 Spec。分级判断准，才不会把规范变成负担。下面四个级别，按"改动的影响半径"由小到大递增。

---

## L0 · 免 Spec（commit message 即记录）

**无外部行为变更**的改动。包括但不限于：

- 拼写 / 注释 / 格式修复
- 日志级别调整（如 INFO → DEBUG）
- 配置值微调（超时时间、阈值等，不改变对外语义）
- 依赖补丁版本升级（无 API 变更，如 `2.3.1 → 2.3.2`）
- 有测试保护且行为不变的纯重构

> 判断要点：**外部调用方 / 使用方完全感知不到任何变化**。commit message 写清楚改了什么即可。

## L1 · 快速 Spec（约 15 行）

**单点行为变更**。包括但不限于：

- Bug 修复（含边界条件遗漏导致的问题）
- 给已有列表 / 组件增加排序、筛选、交互
- 新增单个查询接口（不新增表）

> 判断要点：**只动一个行为点，不引入新的模块或数据结构**。用 L1 快速 Spec 模板（见 `templates/l1-quick-spec.md`）。

## L2 · 标准 Spec

**单系统内的新模块或结构性变更**。包括但不限于：

- 新功能模块（如优惠券、消息通知）
- 新表 / 新字段
- 对接第三方服务
- 权限模型修改
- 一个功能贯穿 Controller + Service + Repository

> 判断要点：**引入了新的"东西"（模块 / 表 / 服务），但仍局限在一个系统内部**。用 L2 标准 Spec 模板（见 `templates/l2-standard-spec.md`）。

## L3 · 完整 Spec

**跨系统或高风险**。包括但不限于：

- 多系统联动
- 微服务拆分
- 更换中间件
- 合规审计链需求
- 对外开放平台 API

> 判断要点：**影响跨越系统边界，或牵涉合规 / 对外承诺**。

---

## 铁律（不可违反）

> **声明纯重构但实际改变了行为 → 退回补 L1。**

"纯重构"是 L0 的通行证，但只要偷偷改了对外行为（哪怕很小，如排序方向从升序变降序），就不再是 L0，至少补一份 L1 快速 Spec 说清楚改了什么。

---

## 决策树（文字版，测验对照用）

```
改动会改变外部行为吗？
├─ 否 → 是纯重构吗？
│       ├─ 是，且有测试保护 → L0
│       └─ 否 → L1（先把"改了什么行为"说清楚）
└─ 是
    │
    └─ 改动半径有多大？
        ├─ 单个行为点（不加模块/表/接口） → L1
        ├─ 单系统内的新模块/结构变更      → L2
        └─ 跨系统/合规/对外 API           → L3
```

## EARS 句式（写验收条件用）

验收条件（AC）用 **EARS** 句式书写，避免"体验要好""速度要快"这类无法测的形容词：

- **WHEN** `<前置条件/触发事件>` **THEN** `<系统应做出的可观测行为>`
- 正常流：`WHEN 用户提交合法订单 THEN 系统返回 200 且订单进入待支付`
- 异常流：`WHEN 订单金额 ≤ 0 THEN 系统返回 400 且不创建订单`
- 边界：`WHEN 查询结果为空 THEN 返回空数组而非 null`

可测的关键：THEN 后面写的是**能被断言的事实**（状态码、返回结构、副作用），不是主观感受。
