# 四个工作方法（一页速查）

> 这四个方法是训练营反复练的核心动作，几乎每关都会用到其中一两个。每个方法配一节，讲清楚"是什么 / 怎么做 / 哪关用到"。

---

## 一、Bugfix 三层描述法

**是什么**：不改代码，先把问题说清楚。把一个 bug 写成三层，AI 才能精准地"只改需要改的"。

**三层**：
1. **当前行为**：现在出问题时，系统实际做了什么（可复现的事实）。
2. **期望行为**：正确情况下，系统应该做什么。
3. **不变量**：这次修复绝不能破坏的已有行为（回归保护线）。

**示例**：
- 当前行为：已取消（CANCELLED）的订单调用 `summarize()` 时，因 `paid_at` 为 None 而抛 TypeError。
- 期望行为：CANCELLED 订单应返回带状态的摘要，不抛错。
- 不变量：PAID 订单的输出格式保持不变。

**用到它的关卡**：c2-09。

## 二、复杂任务四阶段工作流

**是什么**：面对一个 L2 级以上的复杂改动，不要直接让 AI"一把梭"，而是分四阶段走，每阶段留一份产物。

**四阶段**：
1. **头脑风暴（Brainstorming）**：列需求、列约束、列候选方案，留下一份 `brainstorm.md`。允许 AI 给多个方案，你做取舍。
2. **写计划（Writing Plans）**：把选定方案拆成可执行的步骤，留下 `plan.md`。步骤要细到"别人照着做也能完成"。
3. **分步实现（Subagent Dev）**：按计划逐步实现，每步对应一个清晰子任务。可以用 AI 子代理分头干。
4. **对照验收验证（Verification）**：拿写计划时的验收条件（AC）逐条验证实现结果，留下 `verification.md`。不靠感觉，靠 WHEN/THEN 是否成立。

**用到它的关卡**：c3-09、P1 项目。

## 三、AI 审查终止三条件

**是什么**："换个模型再审又能发现新问题"——多轮 AI 审查每轮都可能冒出新意见，若无终止条件就会无限递归。解法不是审到零问题，而是约定什么时候可以停。

**三个终止条件（同时满足才停）**：
1. **①** 连续一轮审查，没有新增任何**真问题**（只有重复或已处理的）。
2. **②** 剩余的意见均为**风格偏好**，不影响正确性与安全。
3. **③** 安全维度、正确性维度的真问题数均为零。

**关键区分**：真问题（影响功能 / 安全 / 数据正确）vs 风格偏好（命名、注释、格式）—— 前者必须修，后者可记入团队风格表后忽略。

**用到它的关卡**：c2-07。

## 四、context-map（上下文映射文件）

**是什么**：让 AI 按"改哪个路径 → 自动加载哪些规则"工作的映射文件。是上下文工程从"手工贴规则"走向"自动加载"的第一步。

**结构**：一份文件，按"文件路径模式 → 该加载哪些规则文件"建立映射。例如：

```yaml
- match: "src/api/**"        # 改 API 层代码时
  load: [rules/api-convention.md, rules/auth.md]
  why: API 必须走鉴权与统一返回体

- match: "src/db/**"         # 改数据层代码时
  load: [rules/db-naming.md]
  why: 表名/字段名遵循部门命名规约
```

**三个要素**（缺一不可）：
- `match`：匹配什么路径（glob 模式）。
- `load`：该自动加载哪些规则文件。
- `why`：为什么这条映射成立（给维护者看，防止日后盲目删改）。

**用到它的关卡**：c1-04。
