# AI Coding Camp：组织级训练体系设计方案

## ——freeCodeCamp 模式的企业化改造（基于 Landing Zone v1.0 现状）

> 版本 v0.1，供讨论迭代。前置输入：《Landing Zone，AI Coding 转型框架》（20260630 反馈整理版）、四小组 53 条问题诊断、30 个动手实验手册、既有 SkillHub 部署、三级验收标准（🔴🟡🟢）。

---

## 0. 一页总览

**一句话方案**：用 freeCodeCamp 的「关卡地图 × 自动判题 × 项目定级」机制做骨架，加上三个企业化关键改造——**毕业条件必须包含组织性产出**、**cohort（训练营）同频节奏**、**判题系统即未来 CI 门禁**——以 12 周为一期，把 Landing Zone 从一套文档资产变成全员的工作方式。

核心公式：

```
认证 = 个人通关（labs 自动判题，freeCodeCamp 式即时反馈）
     + 组织产出（你的 Spec 被跨组评过 / 你的 Skill 被别组用过 / 你评审过别人）
     + 团队场景（2~3 周真实项目 sprint）
```

| 要素 | 设计 |
|---|---|
| 认证体系 | C1 白带·AI 协作开发者 → C2 黄带·规范开发者 → C3 绿带·资产工程师 → C4 黑带·工程教练 |
| 周期 | 12 周一期，每周 3~4 小时，总学时约 40~50h（含项目） |
| 内容底座 | 全部复用 Landing Zone：30 个实验、8 份 SOP、4 类 Spec 模板、6 个 Skill、SkillHub |
| 判题 | CI 自动判题（🔴/🟡）+ LLM rubric 预评 + 人工/互测复核（🟢），沿用三色验收 |
| 组织闭环 | 每期毕业产出直接合入 landing-zone 仓库并升级版本——延续「Workshop 结束即产出」原则 |

## 1. 设计理念

### 1.1 freeCodeCamp 真正可复用的五个机制

freeCodeCamp 的有效性不在「免费」，而在这五个机制：一是**一条线性关卡地图**，学员永远知道下一步做什么，消灭选择成本；二是**微型挑战加即时判题**，每关 5~30 分钟，提交代码跑测试，通过才解锁下一关，反馈延迟为零；三是**项目定级而非考试定级**，证书靠交付项目获得，不靠选择题；四是**学练一体**，几乎没有「课」，讲解嵌在任务卡里，动手占九成；五是**公开进度**，进度对所有人可见，形成温和的同侪压力。这五条全部保留。

### 1.2 为什么不能照搬

freeCodeCamp 培养的是**个人**；而你们的诊断结论恰恰是「个人强、组织弱」——四象限里的 P0/P1 问题（AI 产物评估标准缺失、Spec 规范不统一、知识沉淀失败、全流程脱节）全部是**协作与资产问题**，没有一条是「个人不会写代码」。照搬纯自学模式的结果可以预见：先锋更强、观望者继续观望、资产依旧不沉淀，两极分化（Group 2#9）反而加剧。

### 1.3 三个企业化改造（本方案的灵魂）

**改造一：毕业条件内建「组织性产出」，任何人都无法独自毕业。**

- C2 要求你的 Spec 由**另一个小组**的学员评审通过，且你完成 ≥2 次对他人的评审；
- C3 要求你贡献的 Skill 被**至少一个其他团队**安装使用；
- C4 要求你实际带教一个模块，并交付本部门效能基线报告。

个人能力由此被强制转化为组织连接。这是对「个人强组织弱」最直接的手术。

**改造二：cohort 制而非纯自学。** 自学解决个人成长，同频解决组织成长。12 周固定节奏 + 每周仪式（站会/分享会）+ 中期资产迭代周，让全员在同一时间段讨论同一类问题——这同时把 P2 的「内部常态化分享」直接制度化为课程仪式，不再另起炉灶。

**改造三：判题即门禁。** 每个 lab 的 `.verify/` 判题脚本（测试、静态检查、安全扫描、Spec 检查）不是一次性教具——结营后直接迁移为生产环境的 pre-commit 与 CI 门禁（对应框架 5.4 节）。学员在通关的同时，把 P0「AI 产物评估标准缺失」的基础设施亲手建了出来，并且已经用了 12 周。**训练基建 = 生产基建，先让人习惯尺子，再用尺子量生产。**

### 1.4 与 Landing Zone 的关系

Landing Zone 是**教材与器材**（资产层），Camp 是**教学法与赛制**（运营层）：

```
Landing Zone（是什么 / 用什么）          AI Coding Camp（怎么学会 / 怎么落地）
├── 30 实验、8 SOP、模板、Skill   ──►   关卡与项目的内容来源（不重复建设）
├── SkillHub（已部署）            ──►   C3 的训练场，也是毕业资产的目的地
├── CI 门禁（框架 5.4，待建）     ◄──   判题系统结营后迁移而来
└── 版本 v1.0                     ◄──   每期结营合入产出 → v1.1、v1.2 …
```

## 2. 学员分层与入营

### 2.1 入营摸底（90 分钟，3 题 + 1 卷）

| 题目 | 考察能力域 | 用途 |
|---|---|---|
| T1 环境题：在部门默认工具中加载 rules/skill 并完成一次生成 | 1 / 2 | 是否可免修 C1 前两模块 |
| T2 挑刺题：给一份埋了 8 处缺陷的 Spec 找问题 | 3 / 4 | C2 起点判定 |
| T3 评审题：review 一段含 3 类问题的 AI 生成代码 | 6 / 9 | 分层判定 + 教练苗子识别 |
| 问卷：使用频率、态度、痛点自述 | — | 观望者 / 迷信者识别 |

### 2.2 三类画像与路径

| 画像 | 特征 | 路径 |
|---|---|---|
| 观望者 | 少用或抵触 | C1 全修，与先锋结对；前两周教练重点关注 |
| 使用者 | 日常在用但「野路子」 | T1/T3 通过可免修对应 labs（test-out 挑战免修），把时间花在 C2 及以后 |
| 先锋 | 0.md 作者这类 champion | 直接从 C2/C3 切入，同步进入 C4 教练培养，承担判题校准与跨组评审 |

Test-out 机制是对「个人强」现实的尊重：任何 lab 都可以不学直接提交判题，过了就免修。但**组织性产出无法免修**——评审贡献和资产被采纳没有捷径，这保证了强者的时间被引导到组织最缺的地方。

对「迷信 AI」这一极（Group 2#9 的另一面），把幻觉识别与评估标准前置为 C1/C2 必修，用判题中的「AI 埋雷题」快速校准预期——让他们亲手抓到 AI 的错，比任何说教都有效。

## 4. 认证详细设计

### 4.1 C1 白带 · AI 协作开发者（W1~W2）

关卡复用现有 L1 初级 10 个实验，按部门默认工具出分叉版本（呼应框架 3.1 节的部门默认工具表——A 部门走 Claude Code 版、B 部门走 Qoder 版，判题脚本需双工具兼容，例如 Qoder 需同时验证 `./qoder/rules/` 与 `AGENTS.md` 双生效）。

| 模块 | 关卡（示例命名） | 判题要点 |
|---|---|---|
| M1 环境与工具 | c1-01 部门默认工具安装与配置校验；c1-02 rules/skill 加载验证（对照卡点表） | 脚本自动检测配置文件与加载结果 |
| M2 上下文工程入门 | c1-03 给项目写 AGENTS.md；c1-04 context-map 匹配实验 | 静态检查 + 有无上下文的生成结果对比 |
| M3 第一个 Spec | c1-05 L0/L1 分级判断（10 个改动场景速判）；c1-06 用 EARS 写 Quick Spec | 分级判断自动比对答案；Spec 走 LLM rubric |
| M4 生成与验证 | c1-07 从 Quick Spec 生成代码并跑通测试；c1-08 幻觉初识（代码中预埋 2 处幻觉，找出来）；c1-09 Secret 泄漏体验（故意触发扫描拦截）；c1-10 提交第一个合规 PR | CI 全绿；幻觉题按命中数定色 |

**微项目**（W2，个人，0.5~1 天）：从本组真实 backlog 挑一个 L1 级小需求（Bug 修复或小功能补充），走完 Quick Spec → 生成 → CI → 跨组 Review → 合并全流程。**从第一天起就训战结合，不做玩具题**——这也是控制总学时的关键。

### 4.2 C2 黄带 · 规范开发者（W3~W5）——直击 P0

| 模块 | 关卡 | 判题要点 |
|---|---|---|
| M5 Spec 分级实战 | c2-01 决策树 20 题速判；c2-02 写一份 L2 标准 Spec（功能+技术两件套）；c2-03 「禁止越级」陷阱题（识别伪装成纯重构的行为变更） | rubric：验收条件可测性、异常流覆盖、非功能需求完整度 |
| M6 Spec 评审 | c2-04 给坏 Spec 挑刺（预埋 8 处缺陷）；c2-05 用 Review Checklist 完成一次真实互评 | 命中 ≥6 处为🟡，8 处全中且改写合格为🟢 |
| M7 Code Review 四级法 | c2-06 四维审查实战（安全/正确/性能/可维护各埋 1 雷）；c2-07 「换个模型再审」体验——复现 0.md 描述的无限递归现象，练习用终止条件收敛 | 按维度命中率定色 |
| M8 测试策略 | c2-08 从 AC 生成测试用例（test-generator skill）；c2-09 Bugfix 三层描述法（当前行为/期望行为/不变量 → AI 生成回归测试再修复，即 Kiro Bugfix 模式） | 回归测试先红后绿；不变量未被破坏 |
| M9 安全红线 | c2-10 OWASP 基线扫描 + 依赖漏洞修复实战 | 扫描零高危 |

**认证项目 P1「订单服务」**（W4~W5，2 人结对，跨组评审）：基于现有 `examples/order-service-demo`，完成一个 L2 级新功能（如新增优惠券模块）——L2 Spec 三件套 → Spec 跨组评审 → AI 生成代码与测试 → 四级 Review → CI 门禁 → 合并。产出的 Spec 按 `specs/active → archive` 归档，顺带把框架 Q3/Q4 的存放与防腐机制跑通一遍。

**示例任务卡**（c2-04，供体感）：

```markdown
# c2-04 给这份坏 Spec 挑刺                       ⏱ 30min | 能力域 4

这是一份「新增消息通知功能」的 L2 Spec，作者赶时间，预埋了 8 处
典型缺陷（验收条件不可测 ×2、异常流缺失 ×2、无幂等设计、无回滚
方案、影响范围漏写、与现有 API 约定冲突各 ×1）。
## 任务
1. 用 Review Checklist 逐条审查，行内标注问题
2. 挑其中 3 处改写成合格写法
## 判题
- 自动：标注格式检查
- LLM rubric + 教练抽查：命中 ≥6 处 = 🟡；8 处全中且改写合格 = 🟢
## 你会用到
templates/spec-功能需求.md · SOP-02 编写Spec指南 · SOP-03 AI-Code-Review
```

### 4.3 C3 绿带 · 资产工程师（W7~W9）——直击 P1

| 模块 | 关卡 | 判题要点 |
|---|---|---|
| M10 Skill 工程 | c3-01 解剖一个好 Skill（对照 6 个内置 Skill）；c3-02 你的第一个 Skill（从给定踩坑记录提炼 SKILL.md） | frontmatter/结构自动检查 + **互测**（见下） |
| M11 资产治理 | c3-03 SkillHub 全流程（上传/审批/版本/命名空间）；c3-04 淘汰演习（给 10 个模拟旧资产做去留判定） | 审批流走通；判定与评审团结论的一致度 |
| M12 企业知识接入 | c3-05 MCP 接入 company-docs 并在生成中引用；c3-06 权限与工具白名单配置 | 生成结果可溯源到内部文档 |
| M13 老项目 | c3-07 用 RepoWiki 构建遗留系统上下文；c3-08 遗留代码补测试再重构 | 测试保护下外部行为不变 |
| M14 全流程 | c3-09 Superpowers 四阶段完整走一遍（含 subagent-driven-development）；c3-10 从需求单到发布单的全链路演练 | 各阶段产物齐全且互相引用 |

**互测机制（本方案的关键设计之一）**：c3-02 的 🟢 不由判题脚本给出，而由**另一位学员**按你的 Skill 独立完成指定任务的成功率决定。你的资产好不好，唯一标准是别人用得起来——这是把「组织弱」直接翻译成判题标准。

**认证项目 P2「从踩坑到资产」**（W8~W9，个人或 2 人）：选一个本团队真实反复踩的坑 → 工程化为 1 个 Skill + 1 条 Rule + 1 条 detection-matrix 条目 → 提交 SkillHub 审批 → 推广给至少一个其他团队使用并收集反馈。Rule 条目同步填入框架 4.5 节的部门 rules 表。**每期结营，SkillHub 净增 Skill ≥ 小组数 × 1，且每一个都带真实采纳记录**——SkillHub 从「搭起来了」变成「转起来了」，正好回应「卡点不在搭建，而在于统计平台、团队成员能够刻意用起来」这条现状判断。

### 4.4 C4 黑带 · 工程教练（W7~W12，与 C3 并行）

面向每组 1~2 名先锋/TL/架构师，是规模化的杠杆：首期产出的教练就是第二、三期的师资，从而摆脱「靠外部老师 + 个别 champion」的天花板。

| 模块 | 内容 |
|---|---|
| M15 效能度量 | 六维指标口径（Adoption/Engagement/Acceptance/Delivery Flow/Quality/Cost）、基线采集、月报解读与 ROI 叙事 |
| M16 治理 | 资产周会运营、Spec 三层防腐（archive/CI 检查/月度清理）、Enterprise Policy 策略管控 |
| M17 教学与评审 | 判题标准校准、rubric 一致性练习、反馈方法、答辩主持 |
| M18 变革管理 | 两极分化应对、抵触者转化案例复盘、异地团队带教 |

**认证项目 P3「带教实战」**：二选一——独立带一个 C1/C2 模块的答疑与评审，或主持一个团队场景 sprint；同时交付本部门六维效能基线报告（作为 W12 对比与下期目标的依据）。评审团的运转过程，也是在营内先跑出 0.md#19「主架构师审核机制」争议的答案。

## 5. 认证项目与团队场景库

P1~P3 绑定认证（上文已述）。另设**选修团队场景**，用于 W10~W11 的 2 周 sprint，各组认领一个——延续既有「小组练习场景 2~3 周、每周同步进展」的设计：

| 场景 | 内容 | 主要产出 | 覆盖能力域 |
|---|---|---|---|
| S1 遗留系统改造 | RepoWiki 建上下文 → 补测试 → AI 辅助重构一个真实老模块 | 老项目改造 SOP 补丁 + 改造前后对比数据 | 12 / 13 / 7 |
| S2 跨系统联动 | 订单+库存+通知三系统联动改造（L3 Full Spec，2~3 个小组协作） | 首个 L3 Spec 范例 + 跨团队协作复盘 | 3 / 4 / 13 |
| S3 效能基线冲刺 | 为本部门建立六维指标基线与看板 | 部门基线报告（供 C4 的 P3 复用） | 14 |
| S4 全流程 AI 渗透 | 一个真实需求从需求单到发布全程 AI 辅助 | 全流程 SOP 实测反馈 | 5 / 13 |

场景验收沿用三色标准，场景卡预先写明 🟡/🟢 条件。**S2 建议每期至少一组必选**——跨系统、跨团队正是「组织弱」的终极考场，也是唯一必须多组协同才能通关的场景。

## 6. 判题与验收系统

### 6.1 三色标准量化

沿用 🔴未通过 / 🟡通过 / 🟢优秀，但每关的任务卡必须**预先写明两档的客观条件**（可测、可复核），杜绝「感觉给分」。这本身就是在训练团队定义「评估标准」的能力。

### 6.2 三类判题手段

| 手段 | 适用（占比估计） | 说明 |
|---|---|---|
| CI 自动判题 | 代码类、配置类、扫描类（约 60% 关卡） | push 分支触发 `.verify/`：单测、静态检查、安全扫描、格式校验；freeCodeCamp 式即时反馈 |
| LLM rubric 预评 | Spec 质量、Review 意见质量（约 25%） | 按 rubric 打分并给出预评意见；**教练按 20% 比例抽查校准**，rubric 与人评不一致处每周复盘修订 |
| 人工 / 互测复核 | 🟢 档、Skill 可用性、项目答辩（约 15%） | 双评制（LLM 预评 + C4 复核）；Skill 用互测；项目用跨组评审团 |

### 6.3 判题即门禁：迁移路径

W6 中期迭代周，把 C1/C2 已跑稳的 `.verify` 检查项（Secret Scanning、依赖扫描、lint、Spec 关联检查）打包为 **pre-commit 与 CI 门禁 v0.5**，在自愿仓库试运行；W12 结营发布**门禁 v1.0** 合入 landing-zone；Phase 2 推广到真实业务仓库。到那时，团队对这些检查已有 12 周的肌肉记忆，推广阻力最小。

### 6.4 防「为过而过」

微项目与 P1/P2 一律绑定真实 backlog；🟢 档附带随机口头答辩抽查（教练 10 分钟三问）；排行榜以「评审贡献数、资产被采纳数」为主指标而非通关速度，避免刷题内卷。

## 7. 运营：12 周节奏与角色

### 7.1 周历

| 周 | 主线 | 关键动作 |
|---|---|---|
| W0（并入 Phase 0 筹备） | 开营 | 摸底分层、结对、环境就绪、六维基线采集启动 |
| W1~W2 | C1 | 全员通关 + 微项目；教练重点盯观望者 |
| W3~W5 | C2 | labs + 项目 P1；W5 为跨组评审周 |
| W6 | 中期迭代周 | 复盘卡点热力图；前半程产出（Spec 范例、rules 修订、门禁 v0.5）合入 landing-zone v1.1 |
| W7~W9 | C3（先锋并行 C4） | labs + 项目 P2；SkillHub 提交与互测高峰 |
| W10~W11 | 团队场景 sprint | 各组认领 S1~S4；每周同步进展 |
| W12 | 毕业周 | 答辩 + 资产盘点 + 六维基线对比 + 颁证 + landing-zone v1.2 发布 |

每周仪式：周一 15 分钟小组站会；周五 45 分钟全营分享会（各组轮值供稿，主题即本周踩坑与产出——把 P2「内部常态化分享」变成课程自带仪式）；教练每周 2 次固定 office hours。

### 7.2 角色

| 角色 | 人数 | 职责 |
|---|---|---|
| 营长 | 1 | 整体运营、跨部门协调、与业务侧的工时协议 |
| 教练（C4 学员 / 外部老师） | 每 15~20 学员 1 名 | 答疑、复核、带模块、rubric 校准 |
| 小组长 | 每组 1 | 站会、进度、场景 sprint 负责人 |
| 评审团 | 3~5 人（架构师+教练） | 🟢档复核、项目答辩、SkillHub 资产审批 |
| 平台管理员 | 1 | 判题 CI、Dashboard、SkillHub 命名空间 |

首期的外部老师角色建议从「授课者」转为「总教练」：主要精力放在 M17 判题校准和 C4 带教上，把内容交付交给任务卡和判题系统——这是从「买课」到「造血」的转折点。

### 7.3 异地（南京 / 上海）方案

材料全部 repo 化、任务卡自含讲解（本方案九成靠动手，天然不依赖现场授课）；仪式全线上；各地至少 1 名 C4 教练；场景 sprint 优先同地组队，S2 跨系统场景可刻意安排跨地协作作为进阶挑战。

## 8. 平台与基础设施

### 8.1 课程即代码（Curriculum as Code）

课程本身就放进 landing-zone 仓库，随资产一起版本化、一起治理：

```
landing-zone/
└── camp/
    ├── curriculum.yaml          # 关卡地图：证书→模块→lab 的依赖与解锁关系
    ├── labs/
    │   └── c2-04-spec-review/
    │       ├── README.md        # 任务卡（背景/任务/判题/引用资产）
    │       ├── starter/         # 起始材料（坏 Spec、埋雷代码等）
    │       └── .verify/         # 判题：测试脚本 + rubric.yaml
    ├── projects/                # P1~P3 与 S1~S4 场景卡
    ├── rubrics/                 # 评分细则（Spec 质量 10 项等）
    └── placement/               # 入营摸底三题
```

### 8.2 Dashboard（复用既有统一 Dashboard 平台）

个人视图：关卡进度、段位徽章、评审贡献数、资产被采纳数。组织视图：小组积分、**卡点热力图**（某个 lab 大面积🔴 = 教学设计或工具链问题，直接反哺课程迭代）、部门默认工具登记表（承接框架 3.1 节机制）。数据源：Git 平台 CI 结果 + SkillHub API。

### 8.3 SkillHub 练习区

开设 `camp-<期数>` 命名空间隔离练习资产（SkillHub 原生支持命名空间与审批流）；毕业资产经评审团审批后迁入正式命名空间——审批动作本身就是 SOP-07 的实战检验。

### 8.4 MVP 建设清单（Phase 0 两周内完成）

curriculum.yaml + 前 20 个 lab 的任务卡与判题（C1/C2 优先，约六成可由现有 30 实验直接改造）；CI 判题流水线模板 1 套；LLM rubric 评审器 1 个（先做 Spec 质量）；Dashboard 最小版（进度 + 排行）；摸底三题。预估 2 人 × 2 周——且建设过程本身按 L2 Spec 用 AI Coding 完成，吃自己的狗粮。

## 9. 激励与制度衔接

个人：段位徽章（白/黄/绿/黑）+ 电子证书 + 内网 profile 展示；排行榜主排「贡献指标」（评审数、被采纳数），弱化速度排名。团队：小组积分 = 通关率 ×0.4 + 资产被采纳 ×0.4 + 互评质量 ×0.2，期末设团队奖。

制度衔接（建议项，需与 HR/管理层对齐后生效）：C2 作为参与 AI 重点项目的上岗线；C3/C4 纳入任职资格或晋升材料的加分项；新员工入职默认编入下一期 C1。段位体系同时给 P3 象限「职业规划方向不明」一个组织侧答案：C4 教练/赋能架构师就是 AI 时代工程师的一条新阶梯。

## 10. 度量与 ROI

| 层次 | 指标 | 目标示例 |
|---|---|---|
| 过程 | 周活跃率、关卡通过率、卡点热力图 | 周活 ≥85% |
| 结果 | 各级认证率 | C1 90% / C2 70% / C3 30% / C4 每组 ≥1 |
| 组织（北极星） | **跨团队复用次数**（Skill 被外组安装使用 + Spec 模板被外组引用） | 结营 ≥ 小组数 × 2 |
| 组织 | Spec 覆盖率（PR 关联 Spec 的比例）、Review 发现率、SkillHub 净增与采纳率 | 逐月上升 |
| 业务（六维） | Adoption / Engagement / Acceptance / Delivery Flow / Quality / Cost | W0 采基线，W12 与 W24 两次对比 |

诚实的预期管理：12 周能直接交付的是**能力与资产**（认证率、资产采纳、Spec 覆盖率）；交付效率类指标（Delivery Flow / Quality）通常滞后一到两个季度才显现——这与诊断表引用的行业观察一致（78% 组织用了 AI 但多数提升不到 10%，根因正是只买工具不建体系）。**基线先行，才谈得上 ROI**；北极星选「跨团队复用次数」，因为它是「组织强」最直接的可测代理。

## 附录 B：现有 30 实验的映射与缺口

现有 L1/L2/L3 各 10 个实验按能力域回填至 C1/C2/C3 关卡位（具体映射由课程组对照《30-AI-Coding实验手册》逐一对号，约半天工作量）。预计缺口约 **8~10 个组织协作类新实验**：跨组评审 ×2、Skill 互测 ×2、SkillHub 审批与淘汰演习 ×2、判题校准 ×1、L3 跨系统 Spec ×1、基线采集 ×1。现有实验偏个人操作，组织协作类是空白——这正是「个人强组织弱」在课程资产上的镜像，需在 Phase 0 补齐。

## 附录 C：Spec 质量 rubric（LLM 预评用，10 项示例，满分 10）

背景与问题清晰（1）／用户故事完整（1）／正常流 AC 可测（1.5）／异常流覆盖（1.5）／边界条件（1）／非功能需求填写（1）／影响范围与依赖（1）／DB 与 API 变更声明（1）／回滚考虑（0.5）／与现有约定无冲突（0.5）。≥7 分为🟡，≥9 分为🟢；任一「可测性」项得 0 分直接🔴。
