# 判题机制说明

> 本页讲清楚训练营怎么给你"判分"。读完你就知道为什么有的关只给了🟡、以及🟢从哪来。

## 三色含义

| 颜色 | 含义 |
|------|------|
| 🔴 | **未过**。这一关当前不达标，需要按反馈修改后重新提交。 |
| 🟡 | **通过**。达标了，符合毕业门槛。 |
| 🟢 | **优秀**。不只是达标，产物质量高、可作为范例。 |

## 一条核心规则（务必记住）

> **CI 最高只给 🟡；🟢 永远来自教练或跨组互测。**

为什么？因为 CI 只能检查"形式上对不对"（文件存在、字段齐全、关键词命中），查不出"内容好不好"。一个结构完整但内容空洞的 Spec 能骗过 CI，但骗不过教练的眼睛。所以：

- 机器（CI / 自动脚本）能保证的是**下限**——🟡
- 人（教练复核 / 跨组互测）才能确认**上限**——🟢

这和你以后在工作中把规范引入 CI 是同一个道理：CI 守底线，人守质量。

## 提交与判题流程

1. **建分支**：`camp/<关卡号>/<工号>`（如 `camp/c1-05/10001`）。分支名是判题链路的约定，不要随意改格式。
2. **推送**：把改动推到训练仓库的这个分支。
3. **CI 自动跑**：仓库的 CI 会自动检出你分支里该关的 `.verify/verify.sh` 并执行。
4. **回调平台**：CI 跑完，结果（red / yellow）回调平台，平台记录判分事件并归约颜色。
5. **🟢 看类型**：
   - 如果这关的 🟢 判定方是 **COACH**：教练会在你拿到 🟡 后抽样复核。
   - 如果是 **PEER_TEST / PEER**：平台会指派跨组同学来测你的产物或评审你的提交。

## 本地预跑（强烈推荐，省得来回等 CI）

每次提交前，自己先在本机跑一遍判题脚本：

```sh
cd labs/c1-05-spec-level-quiz   # 进入关卡目录
sh .verify/verify.sh            # 本地预跑
```

脚本通过（`✅`）再推分支，比推上去等 CI 红、改、再推快得多。脚本用的是标准 POSIX sh，macOS / Linux 都能直接跑。

> 首次环境准备（clone 仓库、建分支、第一次推送）见 [labs/c1-00-hello-camp/README.md](../labs/c1-00-hello-camp/README.md)。

> 小贴士：个别关的判题脚本依赖 GNU 工具，Mac 上若报 `command not found`，可用 Docker 跑或问教练。

## 判分记录在哪看

平台「我的关卡」页面会显示你每关的颜色和最近判分事件；教练台和看板能看到全营的判分热力图。所有判分事件都是审计凭证，会长期保留。
