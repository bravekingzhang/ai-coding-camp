# Skill 与 SkillHub 速查

> 本页讲清楚"什么是 Skill""怎么写一个好的 Skill""SkillHub 是什么"。c3-01 ~ c3-04 都用到这些概念。

---

## 什么是 Skill

**Skill = 把一次踩坑经验，工程化成一份"AI 和人都能照着执行"的结构化文档。**

踩坑记录人人会写，但通常是这样：
- ❌ "上周订单导出越用越慢，排查半天搞定了" —— 别人看了还是不会做。

Skill 的要求更高：写到**别人不问你、照着它也能把这件事做完**。它不是"发生了什么"的流水账，而是"怎么做"的执行手册。

---

## SKILL.md 的六段结构

一个合格的 Skill（`SKILL.md`）由六部分组成，每部分都有存在理由：

| 部分 | 回答什么 | 删掉会怎样 |
|------|---------|-----------|
| **frontmatter** | 元数据：名字 / 一句话说明 / 版本 / 负责人 / 标签 | 别人搜不到、不知道找谁、不知道是否最新 |
| **触发时机** | 什么时候该用这个 Skill？ | 该用时想不起来，不该用时误用 |
| **步骤** | 具体怎么做？一步步、可执行 | 看完还是不会做，等于没写 |
| **反例** | 什么做法是错的、为什么不灵 | 别人重复踩你已经踩过的坑 |
| **边界** | 什么情况下这个 Skill 不适用 | 超出适用范围导致翻车 |
| **维护信息** | 谁维护、上次更新时间、依赖什么 | 内容过期没人知道、找不到负责人 |

> 写完后自检：找一个没参与过这件事的同事，让他只看这份 Skill 完成任务。他能做完不追问你，才算合格。

## 什么是 SkillHub

**SkillHub = 企业内部的技能资产平台。** 团队里每个人写的 Skill，沉淀到这里变成全公司可发现、可复用的资产。

SkillHub 的核心机制：

- **命名空间**：每个 Skill 归属一个命名空间（如 `team-payment`、`camp-1` 练习空间），避免重名、便于授权。
- **审批**：Skill 发布前需经过审批，保证进入平台的资产质量达标（不是随手一写就对外）。
- **版本**：每个 Skill 有版本号（如 `1.0.1`），更新走版本递增，使用方可钉版本。
- **安装**：使用方把 Skill"安装"到自己的工作环境，之后就能在自己的 AI 工具里触发使用。

**价值闭环**：一个人踩坑 → 写成 Skill → 审批进 SkillHub → 别的团队"安装"使用 → **同样的坑全公司只踩一次**。这也是 C3「资产工程师」和 P2 项目要打通的全流程。

---

## 一句话区别

- **Rule**：一条"必须 / 禁止"的约定（短、硬、机器可查）。
- **Skill**：一份"怎么做"的执行手册（长、软、给人或 AI 照着做）。
- **Detection**：一条"如何被 CI / 扫描自动拦截"的规则（把人的教训固化进门禁）。

训练营 C3 阶段的目标，就是让你能产出这三类资产，并把它们沉淀进 SkillHub 让全组织受益。
