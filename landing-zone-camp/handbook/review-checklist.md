# 评审清单（Spec 评审 + 代码四维审查）

> 本页是 c2-04 / c2-05 / c2-06 / c2-07 的审查依据。把它当评审时的 checklist 用。

---

## 一、Spec 评审四项

评审一份 Spec（自己写的或别人写的）时，逐项检查：

### 1. 可测性
- 验收条件（AC）是否都用 WHEN…THEN… 句式？
- THEN 后面是**可断言的事实**（状态码、结构、副作用），还是形容词（"体验好""速度快"）？
- ❌ 反例："通知要快" → ✅ 正例："WHEN 触发通知 THEN 系统在 200ms 内发出"

### 2. 异常流覆盖
- 失败路径写了吗？（如：下游超时、数据不存在、权限不足）
- 降级方案写了吗？（如：主流程失败时的兜底行为）
- 权限边界写了吗？（谁能做、谁不能做）

### 3. 非功能需求
- 性能有具体值吗？（不是"要快"，而是"P99 < 200ms"）
- 安全考虑了吗？（鉴权、输入校验、敏感数据脱敏）
- 幂等性说明了吗？（重复调用是否安全、如何保证）

### 4. 回滚方案
- 出问题怎么退回？（开关、灰度、降级路径）
- 回滚会不会丢数据？

## 二、代码四维审查

审查一段代码 / 一个改动时，从四个维度看：

### 1. 安全（Security）
- 有没有 SQL 字符串拼接？（注入风险）
- 有没有把用户输入直接拼进命令 / 路径？（注入 / 路径穿越）
- 敏感信息（密钥、token）有没有硬编码？

### 2. 正确性（Correctness）
- 边界条件对吗？（空值、最大值、越界、off-by-one、分页多/少返回一条）
- 异常分支处理了吗？还是直接崩？
- 有没有假设永远不会失败的地方？

### 3. 性能（Performance）
- 有没有 N+1 查询？（循环里逐条查数据库）
- 有没有会随数据量增长退化的操作？（如 OFFSET 越大越慢的深分页）
- 魔法数字 / 硬编码阈值有没有解释？

### 4. 可维护性（Maintainability）
- 别人接手能看懂吗？（命名、注释、结构）
- 有没有藏得很深的隐式依赖？（全局状态、时序耦合）
- 改起来会不会牵一发动全身？

---

## 评审时怎么区分"真问题"和"风格偏好"

- **真问题**：影响功能、安全、数据正确性 → 必须修，否则不能 🟢。
- **风格偏好**：命名习惯、注释多寡、格式 —— 记入团队风格表即可，不阻塞。

这条区分也是"AI 审查终止三条件"的基础（见 `methods.md`）。
