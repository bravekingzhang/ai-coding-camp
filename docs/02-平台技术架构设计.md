# AI Coding Camp 训练平台：技术架构设计

> 版本 v0.1。配套文档：《AI-Coding-Camp 训练体系设计方案》。本文回答三个问题：用什么框架、系统长什么样、怎么在开营前建出来。

---

## 0. 结论先行

**推荐：Next.js 全栈单体（App Router + TypeScript）+ PostgreSQL/Prisma + Redis/BullMQ 独立 Worker + Tailwind/shadcn-ui，Docker 部署于内网。**（大版本锁定建设启动时的最新稳定版即可，Next.js 15/16 均满足需求。）

比框架更重要的，是一条前提性架构原则：**平台必须是薄的**。

```
Git 仓库   = 课程内容与判题逻辑的唯一事实源（curriculum.yaml、任务卡、rubric、.verify）
CI Runner  = 判题执行器（平台永远不运行学员代码）
SkillHub   = 资产库与采纳数据源（已部署，不重复建设）
LLM 网关   = rubric 预评打分器
─────────────────────────────────────────────
训练平台   = 只做四件事：身份、状态、关系、可视化
```

正因为平台只做这四件事，2~3 人配合 AI Coding 才可能在约 3 周内交付 MVP。任何把判题、内容编辑、代码托管搬进平台的设计，都会把它变成一个做不完的工程。

**备选路线**：若平台的长期归口团队以 Go 为主（与 SkillHub 的 Go+React 同栈），后端换 Go(Gin/Echo) + 前端 React(Vite) 完全成立。本文的领域模型、状态机、API 契约都是栈无关的，换语言只换实现层。

## 1. 选型分析

### 1.1 先回答「要不要自建」

| 方案 | 判断 |
|---|---|
| 改造开源 LMS（Moodle / Open edX） | 不建议。核心机制——Git+CI 判题、跨组评审匹配、资产采纳追踪、认证条件引擎——LMS 一个都没有，二开成本高于自建，且引入庞大无关功能 |
| GitHub Classroom 类 SaaS | 用不了（内网 + 内部 Git 平台），但它「作业 = 仓库、判题 = CI」的模式正是本方案要复刻的对象 |
| 纯 Git + 共享表格（零平台） | 10 人以内可行。50~70 人规模下，跨组评审匹配、三项毕业条件核验、卡点热力图会把营长和教练淹死在人肉对账里 |
| **自建薄平台** | **本方案。约 15 张表 + 4 个外部集成 + 一个状态机，是小而清晰的工程** |

### 1.2 自建前提下的框架对比

| 维度 | Next.js 全栈单体（推荐） | Go + React 前后分离 | Java Spring + Vue |
|---|---|---|---|
| 建设速度（1~2 人） | 最快：一个仓库一种语言，UI 交互走 Server Actions 省掉一层 API 胶水 | 中：两个工程、两套构建、接口联调成本 | 慢：样板代码多，MVP 周期最长 |
| AI 代码生成成熟度 | 最高。Next + Prisma + shadcn/ui 是当前 AI 生成质量与语料最好的组合——**而你们恰恰要用 AI Coding 来建它** | 高 | 中 |
| 与现有技术栈契合 | 前端同学已在用 React 系；平台无重计算（判题在 CI），Node 足够 | 与 SkillHub（Go+React）同栈，若平台组 Go 强则显著加分 | 仅当公司有强制 Java 中台规范时选 |
| 异步任务 | 需独立 Worker 进程（BullMQ），不能塞进 route handler——这是 Next.js 方案唯一要额外交代的点 | goroutine 天然舒适 | 成熟但重 |
| 运维复杂度 | Node 容器 ×2 + Worker ×1，藏在内网 Nginx 后，无 SSR/SEO/边缘等复杂度 | 同级 | 同级偏重 |

**结论：默认 Next.js。唯一改判条件是「平台的长期维护方是一个 Go 团队」。**

对 Next.js 常见顾虑在此场景下逐条不成立：内网系统无 SEO 诉求，不碰 ISR/边缘渲染那套复杂度；并发峰值不过每分钟几十请求，无性能压力；App Router 学习曲线由 AI 生成抹平大半；异步任务用独立 Worker 进程规避 route handler 的短生命周期限制。同时有一条真实收益常被忽略：**平台建设本身就是首批 C2/C3 实战素材**——每个模块先写 L2 Spec、过跨组评审、再让 AI 生成，吃自己的狗粮，建平台的过程就是第一轮教学案例。

### 1.3 技术栈清单

| 层 | 选型 | 理由 |
|---|---|---|
| Web 框架 | Next.js（App Router）+ TypeScript | 全栈单体，见上 |
| UI | Tailwind CSS + shadcn/ui + Recharts | 关卡地图、热力图、排行榜均为图表密集页面 |
| 数据库 | PostgreSQL 16 + Prisma ORM | JSON 字段承载声明式规则；Prisma 的 schema 即文档 |
| 队列/缓存 | Redis 7 + BullMQ（独立 Worker 进程） | LLM 评审、SkillHub 轮询、对账等异步任务 |
| 认证 | Auth.js（NextAuth）对接企业 OIDC/LDAP；试点期可先用工号+验证码 | 企业 SSO 是硬要求，先留好 Provider 抽象 |
| Markdown | remark/rehype + shiki 代码高亮 | 任务卡直接渲染仓库里的 md，内容不进数据库 |
| 通知 | IM 群机器人适配器（企业微信/钉钉/飞书任一） | 通知走 IM 不自建站内信，MVP 只做 webhook 推送 |
| LLM | 统一网关适配器（对接内部模型代理） | rubric 预评；接口抽象为 `grade(rubric, artifact) → scores` |
| 部署 | Docker Compose（MVP）→ 企业 K8s | 镜像入内网 registry，版本全锁定 |

## 2. 总体架构

```
                        内网用户（学员 / 教练 / 组长 / 营长 / 评审团）
                                        │ HTTPS
                                 ┌──────▼──────┐
                                 │ Nginx / 网关 │ ←── 企业 SSO（OIDC）
                                 └──────┬──────┘
                    ┌───────────────────┼────────────────────┐
             ┌──────▼──────┐     ┌──────▼──────┐      ┌──────▼──────┐
             │ Next.js Web │     │ Next.js Web │      │   Worker    │
             │ (页面+API)   │     │  （副本）    │      │  (BullMQ)   │
             └──┬───────┬──┘     └─────────────┘      └──┬───────┬──┘
                │       │                                │       │
         ┌──────▼──┐ ┌──▼──────┐                  ┌──────▼──┐ ┌──▼────────┐
         │Postgres │ │  Redis  │◄─────────────────│ LLM网关 │ │ IM 机器人  │
         └─────────┘ └─────────┘                  └─────────┘ └───────────┘
                ▲  webhook / 回调 / API（全部为适配器）
    ┌───────────┴────────────┬──────────────────────┬────────────────┐
    │ 内部 Git 平台（GitLab等）│ CI Runner（跑 .verify）│  SkillHub API  │
    └────────────────────────┴──────────────────────┴────────────────┘
```

### 四条设计原则

1. **Git 为唯一内容事实源**。curriculum.yaml、任务卡 md、rubric、.verify 全在 landing-zone 仓库；平台通过 main 分支的 push webhook 触发只读同步，数据库里存的是「内容快照 + 学员状态」，不是内容本身。改课程 = 提 MR，课程演进天然有版本、有评审。
2. **判题在 CI，不在平台**。平台永不执行学员代码——安全边界清晰（学员代码可能含业务信息与任意脚本），算力也不归平台管。平台只接收 CI 回调的结构化结果。
3. **一切颜色变化都是事件**。GradeEvent 只追加不修改，最终色 = 事件流归约。判分争议可完整回放，这是认证公平性的技术保障。
4. **集成皆适配器**。Git、LLM、IM 三类外设各定义一个 interface，换 GitLab→Gitea、换模型、换 IM 厂商都不动核心域。

## 3. 领域模型

### 3.1 六个域的划分

| 域 | 实体 | 说明 |
|---|---|---|
| 组织域 | User / UserRole / Cohort / Group / Enrollment | 多期（cohort）多组多角色，一人可跨期复训 |
| 课程域 | CurriculumSync / Lab | Git 同步的只读缓存，Lab 软删防内容漂移 |
| 过程域 | LabAttempt / GradeEvent / RubricEvaluation / ReviewAssignment / PeerTest / Project / ProjectTeam / WeeklySync | 平台的核心：状态与关系 |
| 资产域 | AssetContribution / Adoption | 对接 SkillHub，「被采纳」是 C3 毕业硬条件 |
| 激励域 | CertRule / CertAward / PointsLedger / Badge | 声明式认证规则 + 积分流水 |
| 治理域 | ToolRegistry / PlacementResult / AuditLog | 部门工具登记表、摸底结果、审计 |

### 3.2 核心 Schema（Prisma，可直接落库）

```prisma
// ───────── 组织域 ─────────
model User {
  id          String   @id @default(cuid())
  empId       String   @unique            // 工号，SSO 主键映射
  name        String
  dept        String                      // XX中心XX部门
  site        Site     @default(HQ)
  gitUsername String?  @unique            // 内部 Git 账号，webhook 映射用
  roles       UserRole[]
  enrollments Enrollment[]

model UserRole {
  userId String
  role   Role
  user   User @relation(fields: [userId], references: [id])

enum Role { LEARNER GROUP_LEADER COACH REVIEW_BOARD CAMP_ADMIN PLATFORM_ADMIN }

model Cohort {
  id     String @id @default(cuid())
  name   String                           // "第 1 期"
  startsAt DateTime
  weeks  Int    @default(12)
  status CohortStatus @default(PLANNING)
  groups Group[]
  enrollments Enrollment[]
}
enum CohortStatus { PLANNING RUNNING CLOSED }

model Group {
  id       String @id @default(cuid())
  cohortId String
  name     String                         // 小组1..6 / 南京1 / 上海1
  leaderId String?
  cohort   Cohort @relation(fields: [cohortId], references: [id])
  members  Enrollment[]
}

model Enrollment {                        // 学员在某一期中的存在
  id         String  @id @default(cuid())
  userId     String
  cohortId   String
  groupId    String
  persona    Persona?                     // 摸底画像
  pairWithId String?                      // 结对对象（观望者↔先锋）
  user       User    @relation(fields: [userId], references: [id])
  cohort     Cohort  @relation(fields: [cohortId], references: [id])
  group      Group   @relation(fields: [groupId], references: [id])

}

// ───────── 课程域（Git 只读同步）─────────
model CurriculumSync {

model Lab {
  code       String  @id                  // "c2-04"
  certLevel  Cert

  title      String

  repoPath   String                       // camp/labs/c2-04-spec-review

// ───────── 过程域 ─────────

  id           String @id @default(cuid())
  enrollmentId String
  labCode      String
  status       AttemptStatus @default(LOCKED)

  isTestOut    Boolean @default(false)    // 挑战免修标记
  branch       String?
  mrUrl        String?
  enrollment   Enrollment @relation(fields: [enrollmentId], references: [id])

  id        String   @id @default(cuid())

  source    GradeSource

  payload   Json                          // CI checks 明细 / rubric 分项 / 评语
  actorId   String?                       // 人评时的教练 userId

model RubricEvaluation {
  id         String  @id @default(cuid())

  rubricPath String
  model      String                       // 经 LLM 网关的模型标识
  scores     Json                         // 分项得分
  preColor   Color
  sampled    Boolean @default(false)      // 命中 20% 校准抽样
  coachId    String?
  coachColor Color?
  dispute    Boolean @default(false)      // 人机不一致 → 进 rubric 修订清单
}

model ReviewAssignment {                  // 跨组评审 —— C2 毕业硬条件的载体
  id           String @id @default(cuid())
  cohortId     String
  artifactType ArtifactType
  artifactRef  String                     // attemptId / projectTeamId / mrUrl
  authorId     String                     // enrollmentId
  reviewerId   String                     // enrollmentId，约束：跨组
  status       ReviewStatus @default(PENDING)
  formJson     Json?                      // 按 SOP-03 Checklist 的结构化表单
  quality      Int?                       // 评审质量分（教练抽评，计入积分）
  dueAt        DateTime
  @@index([reviewerId, status])
}
enum ArtifactType { SPEC CODE PROJECT SKILL }
enum ReviewStatus { PENDING SUBMITTED EXPIRED }

model PeerTest {                          // 互测：Skill 的 🟢 由"别人用得起来"决定
  id             String  @id @default(cuid())
  skillAttemptId String
  testerId       String                   // 非作者、跨组
  taskRef        String
  passed         Boolean
  feedback       String?
}

model ProjectTeam {                       // P1~P3 / S1~S4 的期内实例
  id         String @id @default(cuid())
  cohortId   String
  projectCode String                      // "P1" / "S2"
  memberIds  Json                         // enrollmentIds
  finalColor Color?
  syncs      WeeklySync[]
}
model WeeklySync {
  id       String @id @default(cuid())
  teamId   String
  week     Int
  notes    String
  blockers String?
  team     ProjectTeam @relation(fields: [teamId], references: [id])
}

// ───────── 资产域 ─────────
model AssetContribution {
  id           String    @id @default(cuid())
  enrollmentId String
  type         AssetType
  name         String
  skillHubRef  String?                    // SkillHub 命名空间/ID
  approvedAt   DateTime?                  // 评审团审批（走 SkillHub 审批流）
  adoptions    Adoption[]
}
enum AssetType { SKILL RULE DETECTION SPEC_TEMPLATE }

model Adoption {                          // C3 毕业硬条件："被别的团队用过"
  id            String @id @default(cuid())
  assetId       String
  adopterDept   String
  source        AdoptionSource
  evidence      String?                   // 链接 / 说明
  confirmedById String?                   // 采纳方组长确认 —— 防刷关键
  createdAt     DateTime @default(now())
  asset         AssetContribution @relation(fields: [assetId], references: [id])
}
enum AdoptionSource { SKILLHUB_API MANUAL }

// ───────── 激励与治理 ─────────
model CertRule  { certLevel Cert @id  conditions Json }   // 声明式毕业条件
model CertAward { id String @id @default(cuid())  enrollmentId String  level Cert  awardedAt DateTime  evidence Json }
model PointsLedger { id String @id @default(cuid())  enrollmentId String  delta Int  reason String  refType String?  refId String?  createdAt DateTime @default(now()) }
model ToolRegistry { dept String @id  defaultTool String  quirks String? }        // 框架 3.1 部门工具表
model PlacementResult { enrollmentId String @id  t1 Int  t2 Int  t3 Int  persona Persona  exemptions Json }
model AuditLog { id String @id @default(cuid())  actorId String  action String  target String  diff Json?  at DateTime @default(now()) }
```

### 3.3 三个关键设计点

**颜色由事件流归约**：`color = reduce(GradeEvent[])`，规则为「CI 只能给到 🔴/🟡；🟢 按 Lab.greenBy 由教练复核、互测通过或自动加分项给出；后到的人评覆盖机评」。任何判分争议都能逐事件回放——这是把「评估标准」做成制度而非感觉。

**认证条件是数据不是代码**：`CertRule.conditions` 存声明式 JSON，例如 C2 为 `{"labs":"all_yellow","project":{"code":"P1","min":"YELLOW"},"reviews_given":{"gte":2}}`。认证引擎在每次相关事件后增量评估，规则调整（比如下一期把 reviews_given 提到 3）不用改代码、不用发版。

**采纳必须双确认**：SkillHub 的安装数只是线索（source=SKILLHUB_API），计入毕业条件必须有采纳方组长的 confirmedById——「被采纳」是最容易被刷的指标，防刷设计要放在数据模型层而不是靠事后审查。

## 4. 关键流程设计

### 4.1 Git 提交 → CI 判题 → 状态更新（主链路）

```
学员 push 分支 camp/c2-04/{工号}
   │
   ▼ ① Git 平台 webhook（push 事件）
平台 /api/v1/webhooks/git
   → 按分支命名规则映射到 LabAttempt，置 SUBMITTED
   │
   ▼ ② CI 自动触发，执行该 lab 的 .verify/（测试/静态检查/安全扫描）
CI 末步以 pipeline 级短期 JWT 回调平台
   │
   ▼ ③ POST /api/v1/grade/ci-callback
平台落 GradeEvent(source=CI) → 归约出 🔴/🟡
   │
   ├─ verifyType=CI     → 定色，通知学员（IM 机器人），解锁下游关卡
   ├─ verifyType=HYBRID → 入 BullMQ 队列 → ④
   ▼
Worker 取 rubric + 学员产物 → 调 LLM 网关预评
   → 落 RubricEvaluation + GradeEvent(source=LLM)
   → 命中 20% 抽样 → 进教练校准队列 → 教练复核落 GradeEvent(source=COACH)
   → 最终色 → 通知 + 解锁
```

**CI 回调契约**（这是平台与 CI 之间唯一的接口，值得写死）：

```json
POST /api/v1/grade/ci-callback
Authorization: Bearer <pipeline 级 JWT，含 lab/user/commit 声明，有效期 30min>

{
  "lab": "c2-04",
  "emp_id": "10086",
  "commit": "abc1234",
  "pipeline_url": "https://git.internal/.../pipelines/8848",
  "result": "yellow",                  // red | yellow（CI 最高只判到黄）
  "checks": [
    {"name": "secret-scan",   "pass": true},
    {"name": "unit-tests",    "pass": true,  "detail": "12/12"},
    {"name": "spec-linked",   "pass": false, "detail": "PR 未关联 Spec"}
  ]
}
```

兜底：Worker 每 10 分钟对账一次「SUBMITTED 超过 30 分钟无 CI 事件」的 attempt，主动查询 CI API 补录——webhook 丢失不能靠学员截图申诉。

### 4.2 跨组评审匹配（C2 毕业条件的自动化）

Spec/项目提交时触发，约 10 行伪代码即是全部核心逻辑：

```
候选池 = 同期学员
       − 作者同小组成员                    # 硬约束：跨组
       − 最近与作者互评过 2 次的人          # 防固定搭子
       ∩ 进度已达该证书对应模块（如 C2 的 M6）
排序   = (待评任务数 升序, 累计已评数 升序)   # 负载均衡 + 机会均衡
指派   = 取第 1 名，创建 ReviewAssignment(dueAt = +48h)
兜底   = 候选池为空 → 指派给教练并向营长告警
```

评审以结构化表单提交（字段来自 SOP-03 Checklist），而非自由文本——这样评审质量可抽评、可计分、可沉淀为组织的评审语料。

### 4.3 Skill 采纳追踪（C3 毕业条件）

双通道：Worker 每晚轮询 SkillHub API，把正式命名空间下资产的安装/下载增量写入 Adoption(source=SKILLHUB_API)；学员也可手工提交采纳佐证（使用截图/反馈链接）。两者都只有在**采纳方组长在平台上点击确认**后才计入毕业条件。

### 4.4 认证判定引擎

事件驱动：GradeEvent、ReviewAssignment 完结、Adoption 确认三类事件都会触发对应 Enrollment 的规则重评。全部条件满足 → 写 CertAward（evidence 字段快照当时的达成明细，防止事后规则变更引发争议）→ IM 恭喜 + 徽章 + 积分。

## 6. 多用户与多期设计

**角色**：六种角色可叠加（一个先锋可同时是 LEARNER + COACH），权限在中间件按路由组校验，页面内再按数据范围过滤。

**多期租户**：一切过程数据挂在 Enrollment（人 × 期）而非 User 上——第 2 期开营即建新 Cohort，第 1 期数据完整封存；一个人复训、跨期当教练都不产生数据纠缠。CertAward 挂 Enrollment 但认证本身跟人走（查询时按 userId 聚合）。

**并发与一致性**：评审指派用事务 + 唯一约束防止双派；LabAttempt 的 (enrollmentId, labCode) 唯一键保证重复 webhook 幂等；GradeEvent 只追加天然无锁冲突。这个量级（约 200 用户）不存在真正的并发难题，设计上防的是**重复投递**而非高并发。

## 7. 非功能设计

| 项 | 方案 |
|---|---|
| 容量 | 200 学员、峰值每分钟数十请求、LLM 评审全期约 2000 次——Web ×2 副本（发布无感）+ Worker ×1 + PG + Redis，合计约 4C8G 即可 |
| 部署 | docker-compose 四容器起步；后续平移企业 K8s。所有镜像与 npm 依赖入内网源，版本全锁定 |
| 备份 | PG 每日 pg_dump 至内网对象存储/MinIO，保留 30 天；GradeEvent/AuditLog 只增不删 |
| 安全 | webhook 验签；CI 回调用 pipeline 级短期 JWT；平台不存学员代码（只存链接与结构化结果）；LLM 评审经内部网关，产物含业务信息不出内网 |
| 审计 | 一切人工改判、规则变更、采纳确认写 AuditLog——认证公平性靠可追溯，不靠信任 |
| 可观测 | 结构化日志 + 基础指标（队列深度、回调失败率、同步延迟）；MVP 阶段日志够用 |

## 8. 工程结构与建设计划

### 8.1 仓库结构

```
camp-platform/
├── app/                          # Next.js App Router
│   ├── (learner)/                # 学员端：仪表盘/labs/reviews/projects/assets
│   ├── (coach)/coach/            # 校准队列、风险名单
│   ├── (admin)/admin/            # 开营、分组、规则、同步、工具登记
│   └── api/v1/                   # webhooks / ci-callback / adoptions / sync
├── worker/                       # 独立进程（BullMQ）
│   └── jobs/                     # llm-rubric / skillhub-poll / cert-engine / reconcile
├── lib/
│   ├── domain/                   # 状态机、归约器、匹配算法、认证引擎（纯函数，重点单测）
│   ├── adapters/                 # git / llm / im 三个 interface + 各一实现
│   └── auth/
├── prisma/schema.prisma          # 即第 3 章
├── docker-compose.yml
└── e2e/
```

`lib/domain` 是全部业务复杂度所在（约几百行纯函数），也是单元测试的主战场；其余都是胶水。

### 8.2 三周 MVP + 营期内增量

| 阶段 | 交付 |
|---|---|
| 第 1 周 | 骨架 + SSO + curriculum 同步 + 关卡地图/任务卡渲染 + 管理员手工改状态兜底（此时已可开营应急） |
| 第 2 周 | Git webhook + CI 回调 + 状态机 + IM 通知 + 提交详情页 |
| 第 3 周 | LLM rubric Worker + 教练校准队列 + 跨组评审匹配 + 排行/热力图最小版 → **开营线** |
| 营期 W2 | 项目团队页 + 每周同步 |
| 营期 W4 | 摸底录入 + test-out 流程 |
| 营期 W6 | SkillHub 采纳追踪 + 认证引擎（配合中期迭代周上线门禁 v0.5） |
| 营期 W10 | 证书导出 + 徽章墙（毕业周用） |

**不做清单**（与「平台是薄的」互为表里）：不做在线 IDE、不做代码托管与 diff 展示（跳转 Git 平台）、不做站内 IM、不重复 SkillHub 已有的审批/版本/搜索、不做移动原生（响应式即可）、不做 SSR/SEO 优化。

**建设方式**：2 人 × 3 周，全程 AI Coding，且**每个模块先写 L2 Spec、过一次评审、再生成**——平台仓库就是 Camp 的第 0 号示范项目，它的 Spec 与踩坑记录直接成为首期教学素材。

## 9. 风险与对策

| 风险 | 对策 |
|---|---|
| Next.js/依赖版本演进快 | 锁版本 + 内网镜像源；内部系统无追新压力，按年评估升级 |
| webhook 丢失/乱序 | 幂等键 + 10 分钟对账 Job 主动补录（4.1 兜底） |
| LLM 评分漂移 | 建 20 例金标准评测集，rubric 或模型变更必须先跑回归；争议样本进 rubric 修订清单 |
| Worker 单点 | BullMQ 任务持久化于 Redis（AOF），重启续跑；任务全部幂等 |
| 课程内容与学员状态漂移 | Lab 软删 + 同步器对改名/删除给出迁移提示，绝不静默丢状态 |
| 平台被要求「顺便」加功能 | 「不做清单」写进 README，新需求一律先问：这属于身份/状态/关系/可视化四件事吗 |
